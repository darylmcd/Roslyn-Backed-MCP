You are picking up **one or more** initiatives from a previously-produced
backlog-sweep plan (see `backlog-sweep-plan.md`).

- **Serial mode (default):** run one PR, hand off, exit.
- **Parallel mode:** orchestrate 2–4 subagents, each shipping its own PR;
  reconcile state.json + plan.md + shared files centrally.

Mode is decided at Step 2a.

## Preconditions

- Work on a dedicated git branch (or worktree, per `ai_docs/workflow.md`).
- Ship one PR per initiative. The initiative may touch many files; "one concern"
  is the **shared root cause**, not the file count — the plan vetting (Step 1a)
  caps files per initiative, so this naturally holds.
- Close all backlog rows cited by the initiative in the SAME PR (per
  `ai_docs/workflow.md` § Backlog closure) — **serial mode only**. In parallel
  mode, shared-file edits are orchestrator-owned (see the exception below).
- Add a CHANGELOG entry in the SAME PR — **serial mode only**. Parallel-mode
  equivalent: the orchestrator's reconcile PR carries all batch CHANGELOG
  bullets in one edit.
- Do NOT auto-merge unless explicitly authorized by the user for this run.
  Default is hand off at PR-open with validation green.

### Parallel-mode exception — shared files

In parallel mode, `ai_docs/backlog.md` and `CHANGELOG.md` are
**orchestrator-owned**. Subagents MUST NOT edit either file.

Rationale: 2026-04-17 execution passes 1 and 2 each showed 3 of 4 subagents
hit serial-rebase friction because all appended to the same CHANGELOG
subsection (or removed adjacent rows in `backlog.md`). Each rebase cost
1–3 minutes of conflict resolution plus destructive-resolution risk
("accept ours" silently drops a sibling entry).

## Step 1 — Locate current plan

Find the most recent plan directory under `ai_docs/plans/*_backlog-sweep/` (sort by
the timestamp prefix descending). Read its `state.json`. If no plan directory exists,
report "no plan to execute — run `backlog-sweep-plan.md` first" and exit.

To pin a specific plan instead of the newest, the user may pass `plan-id=<dirname>`;
honor that override if supplied.

## Step 1a — Plan vetting (HARD GATE)

Before doing any work, vet the plan against the discipline rules from
`backlog-sweep-plan.md` § Scope discipline. Walk every initiative in `state.json`
and check:

- **Schema version:** `state.json.schemaVersion == 2`. If older, refuse to start —
  emit "Plan was generated by an older planning prompt. Re-run `backlog-sweep-plan.md`
  before executing." and exit.
- **Per-initiative bundling (Rule 1):** any initiative with `rowsClosedCount > 1` must
  have a Diagnosis section in `plan.md` that explicitly cites Rule 1 verification
  (file:line evidence that all 4 conditions hold). If any multi-row initiative lacks
  this, **flag**.
- **Per-initiative file count (Rule 3):** `productionFilesTouched <= 4`. Flag any
  violation.
- **Per-initiative test count (Rule 4):** `testFilesAdded <= 3`. Flag any violation.
- **Per-initiative context budget (Rule 5):** `estimatedContextTokens <= 80000`. Flag
  any violation.

If ANY initiative fails vetting:

- Emit a structured report listing each failing initiative + which rules it violates.
- Set `state.json.vettingFailed = true` and write the failure report to
  `state.json.vettingFailureReport`.
- Exit with the message: "Plan vetting failed. Re-run `backlog-sweep-plan.md` to
  produce a compliant plan, OR confirm with the user that the over-scoped initiatives
  should proceed as-is (executor will then proceed but may exceed budget mid-PR)."

If the user has already confirmed an over-scoped plan can proceed, they may set
`state.json.vettingOverride = true` manually. The executor honors that and proceeds
in serial mode only — parallel mode is disabled when the override is active.

## Step 1b — Reconcile in-review initiatives, finalize if complete

Before picking new work, sync state.json + plan.md against reality. Previous
session(s) opened PRs and exited; some may have merged, some may have been closed
without merge.

**If available, invoke the `/reconcile-backlog-sweep-plan` skill** — it automates
the mechanical parts of this step (gh pr view queries, state.json transitions,
plan.md Status row mirroring, commit on main). Fall through to manual reconcile
only if the skill is unavailable.

For each initiative with `status == "in-review"`:

- `Bash: gh pr view <prUrl> --json state,mergedAt,mergeCommit` (or equivalent).
- **If `state == "MERGED"`:** update state.json (`status → "merged"`,
  `mergedAt → <ISO>`) AND update plan.md's Status row for this initiative to
  `merged (PR #<n>, <YYYY-MM-DD>)`. Commit both edits on `main`.
- **If `state == "CLOSED"` and not merged:** update state.json
  (`status → "deferred"`, `notes → "PR #<n> closed without merge — manual triage
  required"`) and plan.md's Status row to `deferred (PR #<n> closed; see notes)`.
  Surface this in your Step 10 report so the human knows to either reopen + fix,
  or remove from the plan.
- **If `state == "OPEN"`:** leave as `in-review`. Skip.

**After reconciliation, check completion:**

- If every initiative in `state.json.initiatives` now has status `merged`, `obsolete`,
  or `deferred` (i.e. nothing `pending` and nothing `in-progress` and nothing
  `in-review`):
  1. Mark state.json with top-level `completed: true` and `completedAt: <ISO>`.
  2. Add a Refs entry to `ai_docs/backlog.md` pointing at the completed plan,
     matching the existing pattern. Format:
     `| \`ai_docs/plans/{ts}_backlog-sweep/plan.md\` | Backlog sweep ({ts}). Shipped <N> initiatives across <M> PRs; closed <K> backlog rows (P2: <a>, P3: <b>, P4: <c>). |`
  3. Commit these two edits on `main`.
  4. Report "plan {dirname} fully shipped on {date} — run `backlog-sweep-plan.md` to
     start a new sweep" and exit.

Do NOT delete plan.md or state.json. They are the rationale record per the existing
`ai_docs/plans/` convention.

## Step 2 — Select next initiative

From `state.json.initiatives`, filter to `status == "pending"`, sort by `order`
ascending. If no pending initiative exists, report "all initiatives complete" and
exit.

If the first-in-queue has `scheduleHint == "heroic-last"`: verify all other
non-heroic initiatives are `merged` or `obsolete` before proceeding. If not,
skip this one and pick the next pending non-heroic.

## Step 2a — Mode decision (serial vs parallel)

| Condition | Mode |
|---|---|
| User explicitly requested "parallel" / "use subagents" / "batch" | parallel |
| Single initiative selected (only one pending) | serial |
| `state.json.vettingOverride == true` | serial |
| Next-in-queue initiative has `scheduleHint == "heroic-last"` | serial |
| Multiple independent initiatives available AND orchestrator context budget ≥ 100K free | parallel (optional) |

**Parallel-mode picking rules (orchestrator):**

- Select 2–4 initiatives from the pending queue by `order` ascending.
- **Production file sets must be disjoint** across the batch (`ai_docs/backlog.md`
  and `CHANGELOG.md` don't count — orchestrator owns those). Verify by comparing
  each initiative's Scope field in plan.md.
- Each initiative's own `estimatedContextTokens` lives in its subagent's context,
  not the orchestrator's — the orchestrator consumes ~2–5K per spawn for briefing
  + result parsing.

Skip `heroic-last` initiatives until all non-heroic are `merged` or `obsolete`.

## Step 3 — Set up worktree (or branch)

### Serial mode

Create a git worktree at `.worktrees/{initiative.id}/` on branch
`remediation/{initiative.id}` from `main`. (For solo serial execution where worktrees
add overhead, a branch alone is acceptable; the user may instruct simpler ops.)

Update `state.json` (in main checkout, not the worktree):

- `status` → `in-progress`
- `branch` → `remediation/{initiative.id}`
- `worktreePath` → `.worktrees/{initiative.id}` (or null if branch-only)

**Also update `plan.md`'s Status row** for this initiative to
`in-progress (branch: remediation/{initiative.id}, worktree: .worktrees/{initiative.id})`.

Commit BOTH state.json and plan.md edits on `main` in a single commit so concurrent
sessions see this initiative is taken AND the human-readable plan stays in sync.

### Parallel mode

Each subagent creates its OWN worktree inside its prompt (see Appendix B —
Subagent briefing template). The orchestrator does NOT pre-create worktrees
and does NOT pre-update state.json to `in-progress`: that would serialize the
spawn step. The `in-progress` → `in-review` transition is collapsed at Step 9
into one reconcile commit after all subagents return.

Trade-off: during the spawn window, concurrent non-orchestrator sessions
(another human/agent) cannot see the "this initiative is taken" signal.
Acceptable because parallel mode is orchestrator-owned for its duration.

## Step 4 — Verify plan is still accurate

Re-read plan.md section for this initiative. Re-read the source files cited in
Diagnosis. If the behavior has changed since planning (another merge landed):

- **If the initiative is still relevant but the diagnosis shifted:** re-plan in-place
  by updating plan.md with the revised diagnosis/approach. Commit the plan update on
  the worktree branch BEFORE writing code, so the PR shows the re-plan transparently.
- **If the initiative is now obsolete:** set `status → "obsolete"` in state.json on
  main AND update plan.md's Status row to `obsolete (<one-line reason>)`. Commit
  both with a note explaining why, exit. Do NOT skip silently to the next initiative
  — surface the obsolescence so the next planning pass picks up cleaner.

**Re-vet on re-plan:** if you update the Approach to touch more files than originally
estimated (Rules 3–5 violation), STOP. Mark `status → "deferred"`, `notes →
"in-flight scope grew past Rules 3–5 budget; needs re-plan with split"`, exit.

In parallel mode, the orchestrator performs Step 4 for each initiative BEFORE
spawning its subagent. If an initiative is obsolete or needs re-planning, drop
it from the batch and continue with the rest.

## Step 5 — Implement

### Serial mode

Follow the Approach field in plan.md. Rules below.

### Parallel mode — subagent briefing contract

The orchestrator spawns one subagent per initiative via the `Agent` tool using
**Appendix B's briefing template** (below). Key mandates the template enforces:

1. **Cold-context preview-evidence caveat.** A fresh subagent has zero prior
   conversation turns. The PreToolUse hook (`hooks/hooks.json`) that guards
   `mcp__roslyn__*_apply` tools looks for prior `*_preview` evidence in the
   conversation and blocks when none is found. PR #230's widening to accept
   `apply_composite_preview` redemption does NOT help a cold subagent — it
   has no redemption to point to either. The orchestrator's prompt MUST choose
   one of:

   - **Edit-only path** (default for ≤3-file edits): subagent uses
     `Edit`/`Write`/`Read` for mutations. Read-side MCP tools
     (`symbol_search`, `find_references`, `document_symbols`, `compile_check`,
     `test_run`, `test_related_files`, `project_diagnostics`) are always safe
     and preferred over `Grep` / `Bash: dotnet build`.

   - **Preview-then-apply path** (solution-wide rename/extract): subagent
     calls `*_preview` before each `*_apply`. The in-conversation preview
     redeems at the same turn.

   Pre-classify which path each initiative should take in the plan.md Approach
   field so spawns can be templated.

2. **Shared-file discipline.** Subagent MUST NOT touch `ai_docs/backlog.md`
   or `CHANGELOG.md`. Orchestrator handles both in Step 7 (parallel variant).

3. **Result-format contract.** See Appendix C. Subagent's final message MUST
   start with `<<<RESULT>>>` on its own line followed by `success:` + PR URL
   + file list, or `failure:` + blocker. Any deviation is treated as failure.

### Write-side (worktree rules — same for both modes)

Backlog-sweep executor sessions run inside a `.worktrees/<id>/` worktree against the
**installed global tool** at `%USERPROFILE%\.dotnet\tools\roslynmcp.exe`. The running
MCP server binary is a distinct artifact that is NOT mutated by edits to the worktree
source. The main-checkout `bootstrapCaveat` rationale ("binary under edit = binary
servicing the call") does not hold here, so:

- ✅ `*_apply` is safe and preferred when a Roslyn MCP refactor tool covers the
  operation (rename, extract/move type, code fix, bulk type replace, organize usings,
  format) — **in serial mode or in the parallel Preview-then-apply path**. Load the
  worktree's own `RoslynMcp.slnx` first; follow up with `workspace_reload` if a
  downstream call needs a refreshed snapshot.
- ✅ `*_preview` — use first; review the diff; then apply.
- ✅ `Edit` / `Write` — fine for cases the Roslyn MCP surface doesn't cover, or when
  the diff is small and textual. **Default choice for parallel-mode Edit-only path.**

See `ai_docs/runtime.md` § *Bootstrap scope — self-edit on THIS repository* for the
two-sub-case policy (main-checkout restriction vs worktree carve-out).

### Read-side (always safe — including bootstrap mode — 5–30× faster than Bash/Grep)

Before reaching for `Grep` or `Bash: dotnet build`, consult
`ai_docs/bootstrap-read-tool-primer.md` for the pattern→tool table.
Highest-leverage substitutions for this step:

- **Compile after each meaningful edit:** `mcp__roslyn__compile_check` on the loaded
  workspace. Returns structured diagnostics in <1s. Only fall back to
  `Bash: dotnet build RoslynMcp.slnx -c Release -p:TreatWarningsAsErrors=true` if the
  server is disconnected or the failure class specifically requires MSBuild
  (e.g. analyzer-package resolution bugs).
- **Test after each meaningful edit:** `mcp__roslyn__test_related_files` on the files
  you touched to derive the targeted filter, then `mcp__roslyn__test_run --filter`.
  Full-suite `Bash: dotnet test` is only correct for the Step 6 CI-parity run.
- **Find callers / consumers / implementations:** `mcp__roslyn__find_references` with
  `metadataName` (or `filePath+line+column`) — not `Grep` for the simple name. Grep
  matches `SymbolFinder.FindReferencesAsync` when you wanted
  `MyService.FindReferencesAsync`.
- **Find a symbol by name (simple / fuzzy / FQN):** `mcp__roslyn__symbol_search` —
  not `Grep`. Handles `"SampleLib.Dog"` and `"SampleLib.AnimalService.CountAnimals"`
  natively since v1.19.2.
- **Enumerate a file's public surface:** `mcp__roslyn__document_symbols` — not
  `Grep` for `public `.

### Validation rigor

- Add or update tests per the Validation field. Tests MUST cover the specific
  regression (not just "the class still works"). One regression test per closed
  backlog row is the floor; more if the row covers multiple symptoms.

### Parallel mode — orchestrator-owned, separate reconcile PR

**Subagents do NOT edit `ai_docs/backlog.md` or `CHANGELOG.md`.** Their PRs
contain only the initiative's production + test changes. After all subagents in
the batch return AND their PRs merge, the orchestrator opens ONE reconcile PR
that:

- Removes all closed rows from `ai_docs/backlog.md` (alphabetical integrity
  preserved).
- Bumps `updated_at:` once.
- Adds all CHANGELOG bullets under `[Unreleased]` — one edit, no conflicts.
- Updates Maintenance tally atomically (rows-closed counts + test counts).
- Updates state.json `status` → `merged` + `prUrl` + `mergedAt` for each.
- Updates plan.md Status rows for each.

PR title pattern: `chore(plan): reconcile batch-{n} (initiatives X, Y, Z merged)`.
Precedent: PRs #233, #235, #240 in the 2026-04-17 execution passes.

## Step 8 — Open PR

Use the `ship` skill if available, otherwise `Bash: gh pr create ...` and
`gh pr merge`.

### Worktree-cd discipline (both modes)

`gh pr merge` (and auto-merge flags) fails inside a worktree with
`fatal: '<branch>' is already used by worktree at <primary-path>` because `gh`
tries to update / delete the local branch which is checked out in the primary
repo path. Always run `gh pr merge` commands from the primary repo root:

```
cd "$(git rev-parse --git-common-dir)/.."
gh pr merge <n> --squash --auto
```

Or teach ship-style skills to detect worktree context and auto-cd.

### PR title + body

PR title: `{type}({scope}): {short description} ({initiative.id})`. Examples:
`fix(refactoring): create_file_apply no longer reserializes unrelated csprojs (create-file-apply-csproj-side-effect-all-projects)`.

PR body must include:

- **Closes:** explicit list of backlog row ids closed (machine-greppable).
- **Validation:** verify-release.ps1 outcome, test counts (before → after), any
  notable manual checks.
- **Performance:** before/after numbers if Step 4 of plan declared a perf review.
- **Spin-offs:** any new backlog rows added during verification (Step 3 of the plan).

## Step 9 — Update state

### Serial mode

In the main checkout (not the worktree), update `state.json` for this initiative:

- `status` → `in-review`
- `prUrl` → `<url>`
- `notes` → any caveats for the human reviewer

**Also update `plan.md`'s Status row** for this initiative to `in-review (PR #<n>)`.

Commit BOTH state.json and plan.md edits on `main` in a single commit (separate
commit, outside the worktree). The PR-merge → `merged` status transition happens at
the next session's Step 1b reconciliation pass — not here.

### Parallel mode

Orchestrator collects all subagent results, **verifies each PR URL via
`gh pr view <n> --json state,mergeable`** before trusting the claim (see
Appendix C). Batches state.json + plan.md updates into the single reconcile
PR from Step 7 parallel variant. Collapsed state transition:
`pending → merged` in one commit (skipping the `in-progress`/`in-review`
intermediate states that serial mode uses to signal other agents).

### Post-merge main-sync idiom (both modes, every reconcile/ship)

After `gh pr merge --squash` on a PR whose branch carries a local commit,
`git pull --ff-only` on main may refuse or silently no-op — the squash creates a
fresh commit SHA on origin unrelated to your local commit. Use:

```
git fetch origin && git reset --hard origin/main
```

Do this every time after a reconcile-PR merge, or whenever local main diverges
from origin/main post-squash. This is non-destructive for origin-tracked
content; only your ephemeral local commit is discarded (which is correct —
it's been superseded by the squash commit on origin).

## Step 10 — Report and exit

### Serial mode

Output a one-paragraph summary:

- Initiative id
- Rows closed (count + list)
- Files touched (count)
- PR URL
- CHANGELOG section the entry landed in
- Anything unusual the reviewer should look at first
- Whether the next-in-queue initiative is runnable without human input, or blocked

Then **exit**. Do NOT start the next initiative in the same session — context

(Exception: if the user has explicitly authorized a multi-initiative run in this
session, continue to Step 1b reconciliation for the next pending initiative. The
context-cost-per-initiative budget from the plan still applies — if the running total
of session context approaches 80% of the model's window, stop after the current
initiative even under multi-run authorization.)

### Parallel mode

Output a one-paragraph summary covering the batch:

- Initiative ids shipped in this batch (with their PR numbers)
- Reconcile PR number + URL
- Any subagent failures + retry / defer decisions
- Worktree cleanup outcome (count removed, any stragglers)
- Context budget remaining (for deciding whether to run another batch)

If more pending initiatives remain AND orchestrator context ≥ 60% free AND no
heroic-last is next in queue: offer to run another batch. Otherwise, hand off
and exit.

## Appendix A — Parallel-mode orchestrator playbook

Compact checklist for the orchestrator. All steps run from the primary repo root
unless otherwise noted.

1. Steps 1 / 1a / 1b (use `/reconcile-backlog-sweep-plan` skill if available).
2. Steps 2 + 2a — pick 2–4 pending initiatives with disjoint file sets.
3. Spawn subagents concurrently — single assistant message with multiple `Agent`
   tool uses, each with `run_in_background: true`. Briefing per Appendix B.
4. **Do not poll.** The harness notifies on each subagent's completion.
5. Parse each returned result per Appendix C. Verify PR URLs via `gh pr view`.
   On failure, decide retry (tightened brief) vs defer (`status: "deferred"`).
6. Merge each subagent's PR in turn. Rebase-on-merge conflicts on
   `CHANGELOG.md` / `backlog.md` indicate subagent broke shared-file discipline
   — fix that subagent's prompt for future batches.
7. After all batch PRs merge: open ONE reconcile PR per Step 7 parallel variant,
   merge it, run the post-merge main-sync idiom.
8. Worktree cleanup:
   - `git worktree remove --force .worktrees/<id>` for each.
   - `git push origin --delete remediation/<id>` for each merged branch.
   - `git remote prune origin`.
   - `rm -rf .worktrees/` if empty.
9. Step 10 parallel-mode exit.

## Appendix C — Subagent result protocol

Orchestrator parses each subagent's final message:

1. **Locate the `<<<RESULT>>>` sentinel.** Absent → mark failure.
2. **Next non-blank line** starts with `success:` or `failure:`.
3. **Success path:** extract PR URL. Verify via
   `gh pr view <n> --json state,mergeable` — if `state` is not `OPEN`
   or `MERGED`, treat as failure (subagent may have hallucinated the URL —
   observed 2026-04-17 Init 17 first attempt).
4. **Failure path:** record the blocker reason. Decide retry (with tightened
   brief — usually switch from preview-then-apply to edit-only, or narrow
   the scope) or defer (`status: "deferred"` in state.json with the blocker
