## Item 5 — `severity-medium-breaks-msbuild-until-csproj-is-hand`

**Backlog summary:** `create_file_apply` injects explicit `<Compile Include="…">` on SDK-style csproj → MSBuild error `Duplicate 'Compile' items` after `workspace_reload`; breaks `dotnet build` until hand-edited.

| Field | Content |
|-------|---------|
| **id** | `severity-medium-breaks-msbuild-until-csproj-is-hand` (also closes `dr-9-1-add-items-that-break-sdk-style-projects`, `dr-9-6-bug-compile-include-adds-explicit-on-sdk-auto-in`) |
| **Diagnosis** | Verified. `FileOperationService.cs:48` calls `project.AddDocument(filename, content, folders, fullPath)`. `InterfaceExtractionService.cs:104` and `TypeExtractionService.cs:92` do the same. Persistence at `RefactoringService.cs:533–538` calls `_workspace.TryApplyChanges(workspaceId, modifiedSolution)`. **MSBuildWorkspace.TryApplyChanges**, when it sees an `AddedDocument`, mutates the project XML to add an explicit `<Compile Include="…"/>` regardless of whether the project is SDK-style with `EnableDefaultCompileItems=true`. Because SDK-style projects already implicitly include the file via the default glob, this produces a duplicate Compile item. There is no SDK-detection guard in our code — `RefactoringService.cs:533` calls `TryApplyChanges` unconditionally. After `workspace_reload`, MSBuild evaluates the duplicate and errors. **Tests:** `FileOperationIntegrationTests.cs` does not assert "csproj XML unchanged after create_file_apply on SDK-style project." |
| **Approach** | (1) Add a helper `IsSdkStyleWithDefaultCompileItems(Project project) → bool` in `src/RoslynMcp.Roslyn/Helpers/MSBuildProjectInspector.cs`. (2) In `RefactoringService.PersistDocumentSetChangesAsync` (~line 533), if the project IS SDK-style with default Compile items, write the new files to disk directly (`File.WriteAllTextAsync` to the `fullPath`) and SKIP `TryApplyChanges`'s csproj mutation. (3) For non-SDK-style projects (legacy csproj), keep the current `TryApplyChanges` path — explicit Compile items are required there. (4) For Item #2's revert flow: when the new path is taken (no csproj mutation), the FileSnapshot needs only file-content restoration; no csproj-snapshot needed in that case. |
| **Scope** | Modified: `RefactoringService.cs` (PersistDocumentSetChangesAsync conditional path), new helper `MSBuildProjectInspector.cs` (single static method), all `*_apply` paths that go through it (FileOperationService, InterfaceExtractionService, TypeExtractionService, TypeMoveService) inherit the fix without further change. Estimated 1 prod file modified, 1 helper file new, 1 test file modified, 0 deletes. |
